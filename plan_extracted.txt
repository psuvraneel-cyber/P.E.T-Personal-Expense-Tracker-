1. Executive Summary 
Issue 1 — All UPI Transactions Show 12:00 AM 
The Transaction History screen (confirmed in the provided screenshot) shows every SMS-
detected transaction timestamped at 12:00 AM, regardless of when the transaction 
actually occurred. This is a data-mapping bug introduced during the conversion of the 
Android date_sent SMS field (a Unix millisecond timestamp) into a Dart DateTime object. 
The time component is being truncated or discarded — leaving only the date portion, 
which defaults to midnight (00:00:00). 
Impact: Affects all auto-detected transactions. Users cannot determine when during the 
day a transaction occurred. The timeline view (“Yesterday”, grouping) is correct but the 
time within each group is wrong. 
Issue 2 — No Delete Account Option 
The app has only a “Sign Out” option (Clear profile and sign out). There is no 
mechanism for users to permanently delete their account and all associated data. This is: - 
A DPDP Act 2023 compliance requirement in India (the right to erasure) - A Google 
Play Store requirement (as of May 2023, apps with account creation must offer account 
deletion) - A user trust issue — users concerned about financial data privacy expect this 
option

2.1 What the Screenshot Tells Us

Every single transaction in the “Yesterday” group shows 12:00 AM. This is not a 
coincidence — this is a systematic timestamp truncation bug affecting all SMS-sourced 
transactions. 
2.2 The SMS Data Path 
The Android SMS inbox provides two time fields: 
Android SMS Field Type Contains 
date Long (milliseconds) Time the SMS was received by the device 
date_sent Long (milliseconds) Time the SMS was sent by the server 
Both are Unix epoch milliseconds (e.g., 1741804800000 = 2026-03-12 12:00:00 UTC). 
The confirmed field date_sent is referenced in the app’s binary, suggesting it is the 
intended source for transaction timestamps.

2.4 Root Cause — Three Candidate Bugs 
Based on the schema (timestamp stored as TEXT NOT NULL) and the symptom, there are 
three likely root causes. The most probable is Bug A: 
 
Bug A (Most Likely): Date-only string stored, time component dropped 
The date_sent value is converted to a date string using a date-only format like 'yyyy-MM-
dd' instead of a full datetime format. When the stored string "2026-03-12" is later parsed 
back to a DateTime, it defaults to DateTime(2026, 3, 12) which is midnight — showing 
12:00 AM. 
// ❌ BUG — What is currently happening (inferred): 
final dateSentMillis = map['date_sent'] as int; 
final dateTime = DateTime.fromMillisecondsSinceEpoch(dateSentMillis); 
final dateString = DateFormat('yyyy-MM-dd').format(dateTime); // TIME LOST 
HERE 
// Stored: "2026-03-12" → when parsed back → 2026-03-12 00:00:00 → "12:00 AM" 
 
Bug B (Possible): UTC vs Local time confusion causing midnight 
The date_sent is a UTC millisecond timestamp. If it’s parsed without .toLocal(), and the 
display formatter assumes local time but gets UTC midnight, transactions sent at exactly 
midnight UTC (8:00 AM IST) could appear as “12:00 AM”. However, this would only affect a 
subset of transactions — not all of them — so this alone doesn’t explain all showing 12:00 
AM. 
 
Bug C (Possible): Fallback to date-only when date_sent is missing 
When date_sent is 0 or null (which happens for some SMS types), the parser may fall back 
to the date extracted from the SMS text body, which is date-only. This would affect all UPI 
transactions if the SMS body date is being used instead of the system timestamp. 
// ❌ BUG — Fallback uses body-parsed date (date only, no time): 
final timestamp = map['date_sent'] != null && map['date_sent'] != 0 
    ? DateTime.fromMillisecondsSinceEpoch(map['date_sent']) 
    : _parseDateFromSmsBody(map['body']); // This returns date-only → 
midnight 
2.5 Confirmed Schema Evidence 
From the binary analysis: 
sms_transactions table fields: 
  timestamp TEXT NOT NULL          ← Stored as string, not INTEGER 
  date TEXT NOT NULL               ← Separate date-only field also exists 
The existence of two separate fields — timestamp (full datetime) and date (date-only) — 
strongly suggests a design issue where the date field (date-only, no time) is being used for 
display instead of the timestamp field. This is Bug A expressed as a field selection error.

4.1 native_sms_reader.dart — Capture Full Millisecond Timestamp 
// lib/services/native_sms_reader.dart 
 
class NativeSmsMessage { 
  final String id; 
  final String address;    // SMS sender (e.g., "HDFCBK", "8479054125@ybl") 
  final String body; 
  final int dateSentMillis; // ← Rename from any date-only field 
  final int dateReceivedMillis; 
 
  NativeSmsMessage({ 
    required this.id, 
    required this.address, 
    required this.body, 
    required this.dateSentMillis, 
    required this.dateReceivedMillis, 
  }); 
 
  factory NativeSmsMessage.fromMap(Map<dynamic, dynamic> map) { 
    // Android SMS Telephony columns: 
    // Telephony.Sms.DATE      → time received by device (millis since epoch) 
    // Telephony.Sms.DATE_SENT → time sent by server    (millis since epoch) 
     
    final dateSent = map['date_sent'] as int? ?? 0; 
    final dateReceived = map['date'] as int? ?? 0; 
     
    // Use date_sent if available and non-zero, fall back to date (received) 
    // NEVER fall back to a date-only string from the SMS body for timestamp 
    final effectiveTimestampMillis = (dateSent > 0) ? dateSent : 
dateReceived; 
     
    return NativeSmsMessage( 
      id: map['_id']?.toString() ?? '', 
      address: map['address']?.toString() ?? '', 
      body: map['body']?.toString() ?? '', 
      dateSentMillis: effectiveTimestampMillis, 
      dateReceivedMillis: dateReceived, 
    ); 
  } 
 
  /// Full DateTime including time-of-day, in local timezone 
  DateTime get transactionDateTime => 
      DateTime.fromMillisecondsSinceEpoch(dateSentMillis).toLocal(); 
   
  /// ISO 8601 string with time: "2026-03-12T14:35:00.000" 
  /// This is what should be stored in sms_transactions.timestamp 
  String get isoTimestamp => transactionDateTime.toIso8601String(); 
}

// lib/screens/sms_transactions/sms_transactions_screen.dart 
 
/// Format a stored timestamp string for display 
String _formatTransactionTime(String? timestamp) { 
  if (timestamp == null || timestamp.isEmpty) return '—'; 
   
  try { 
    final dt = DateTime.parse(timestamp).toLocal(); 
     
    // Check if time is midnight — likely a legacy truncated record 
    if (dt.hour == 0 && dt.minute == 0 && dt.second == 0) { 
      // Show date only for legacy records (avoid misleading "12:00 AM") 
      return DateFormat('d MMM').format(dt); 
    } 
     
    // Normal case: show time in 12-hour format (e.g., "2:35 PM") 
    return DateFormat('h:mm a').format(dt); 
  } catch (e) { 
    return '—'; 
  } 
}

Future<void> _migrateTimestamps(Database db) async { 
  // Mark all existing sms_transactions with midnight timestamps 
  // as "approximate" so the UI can show them differently 
  await db.execute(''' 
    ALTER TABLE sms_transactions  
    ADD COLUMN IF NOT EXISTS timestamp_is_approximate INTEGER DEFAULT 0 
  '''); 
   
  // Flag all records currently stored with midnight timestamps 
  // ISO format midnight: ends in "T00:00:00.000" or "T00:00:00" 
  await db.rawUpdate(''' 
    UPDATE sms_transactions  
    SET timestamp_is_approximate = 1 
    WHERE timestamp LIKE '%T00:00:00%' 
       OR LENGTH(timestamp) = 10 
  '''); 
   
  debugPrint('[Migration] Flagged approximate timestamps on existing SMS 
records.'); 
}

    const val ADDRESS = "address" 
    const val BODY = "body" 
    const val DATE = "date"           // Time received by device (millis) 
    const val DATE_SENT = "date_sent" // Time sent by server (millis) ← MUST 
include this

fun readSmsMessages(context: Context, sinceMillis: Long): List<Map<String, 
Any>> { 
    val messages = mutableListOf<Map<String, Any>>() 
    val uri = Telephony.Sms.Inbox.CONTENT_URI 
     
    val projection = arrayOf( 
        SmsColumns.ID, 
        SmsColumns.ADDRESS, 
        SmsColumns.BODY, 
        SmsColumns.DATE,        // ← receiving timestamp 
        SmsColumns.DATE_SENT,   // ← sending timestamp (prefer this) 
        SmsColumns.READ, 
    ) 
     
    val cursor = context.contentResolver.query( 
        uri, projection, 
        "${SmsColumns.DATE} > ?", 
        arrayOf(sinceMillis.toString()), 
        "${SmsColumns.DATE} DESC" 
    ) 
     
    cursor?.use { c -> 
        while (c.moveToNext()) { 
            messages.add(mapOf( 
                "_id"       to 
c.getLong(c.getColumnIndexOrThrow(SmsColumns.ID)), 
                "address"   to 
(c.getString(c.getColumnIndexOrThrow(SmsColumns.ADDRESS)) ?: ""), 
                "body"      to 
(c.getString(c.getColumnIndexOrThrow(SmsColumns.BODY)) ?: ""), 
                "date"      to 
c.getLong(c.getColumnIndexOrThrow(SmsColumns.DATE)), 
                "date_sent" to 
c.getLong(c.getColumnIndexOrThrow(SmsColumns.DATE_SENT)), // ← KEY 
            )) 
        } 
    } 
    return messages 
}

5.3 Security Requirements for Account Deletion 
1. Re-authentication required — Before deletion, the user must re-authenticate 
(Google Sign-In confirmation). This prevents accidental deletion and protects 
against stolen sessions. 
2. Explicit confirmation dialog — Two-step: “Delete Account” button → confirmation 
modal with typed confirmation or checkbox. 
3. Clear data disclosure — Show exactly what will be deleted before the user 
confirms. 
4. Irreversibility warning — Make absolutely clear this cannot be undone. 
5. Deletion audit trail — Do NOT store deletion logs that contain PII. Log only the 
event (timestamp + anonymised ID hash) for regulatory proof of compliance.

7.1 account_deletion_service.dart — New File 
// lib/services/account_deletion_service.dart 
 
import 'package:cloud_firestore/cloud_firestore.dart'; 
import 'package:firebase_auth/firebase_auth.dart'; 
import 'package:shared_preferences/shared_preferences.dart'; 
import 'package:sqflite/sqflite.dart'; 
 
enum DeletionStep { 
  clearingLocalData, 
  deletingCloudTransactions, 
  deletingCloudBudgets, 
  deletingCloudCategories, 
  deletingCloudPremiumData, 
  deletingUserProfile, 
  deletingAuthAccount, 
  clearingPreferences, 
  complete, 
} 
 
class AccountDeletionService { 
  final FirebaseFirestore _firestore; 
  final FirebaseAuth _auth; 
  final DatabaseHelper _dbHelper; 
 
  AccountDeletionService({ 
    FirebaseFirestore? firestore, 
    FirebaseAuth? auth, 
    required DatabaseHelper dbHelper, 
  })  : _firestore = firestore ?? FirebaseFirestore.instance, 
        _auth = auth ?? FirebaseAuth.instance, 
        _dbHelper = dbHelper; 
 
  /// Stream of deletion progress for UI display 
  final _progressController = StreamController<DeletionStep>.broadcast(); 
  Stream<DeletionStep> get progress => _progressController.stream; 
 
  /// Re-authenticate the user before deletion (security requirement) 
  Future<bool> reAuthenticate(AuthCredential credential) async { 
    try { 
      await _auth.currentUser?.reauthenticateWithCredential(credential); 
      return true; 
    } on FirebaseAuthException catch (e) { 
      debugPrint('[AccountDeletion] Re-auth failed: ${e.code}'); 
      return false; 
    } 
  } 
 
  /// Execute the full account deletion sequence. 
  /// Must be called AFTER successful re-authentication. 
  Future<void> deleteAccount() async { 
    final user = _auth.currentUser; 
    if (user == null) throw StateError('No authenticated user'); 
 
    final uid = user.uid; 
 
    try { 
      // ── Step 1: Clear local SQLite data 
────────────────────────────────── 
      _progressController.add(DeletionStep.clearingLocalData); 
      await _clearLocalDatabase(); 
 
      // ── Step 2: Delete Firestore transactions 
───────────────────────────── 
      _progressController.add(DeletionStep.deletingCloudTransactions); 
      await _deleteFirestoreCollection(uid, 'transactions'); 
 
      // ── Step 3: Delete Firestore budgets 
────────────────────────────────── 
      _progressController.add(DeletionStep.deletingCloudBudgets); 
      await _deleteFirestoreCollection(uid, 'budgets'); 
 
      // ── Step 4: Delete Firestore categories 
─────────────────────────────── 
      _progressController.add(DeletionStep.deletingCloudCategories); 
      await _deleteFirestoreCollection(uid, 'categories'); 
 
      // ── Step 5: Delete premium data collections 
─────────────────────────── 
      _progressController.add(DeletionStep.deletingCloudPremiumData); 
      for (final collection in [ 
        'saving_goals', 
        'recurring_payments', 
        'alerts', 
        'family_members', 
        'linked_accounts', 
        'tax_categories', 
      ]) { 
        await _deleteFirestoreCollection(uid, collection); 
      } 
 
      // ── Step 6: Delete user profile document 
────────────────────────────── 
      _progressController.add(DeletionStep.deletingUserProfile); 
      await _firestore.collection('users').doc(uid).delete(); 
 
      // ── Step 7: Delete Firebase Auth account 
────────────────────────────── 
      // This MUST come after all Firestore operations, because after 
deletion 
      // the user will no longer be authenticated and cannot access 
Firestore. 
      _progressController.add(DeletionStep.deletingAuthAccount); 
      await user.delete(); 
 
      // ── Step 8: Clear SharedPreferences 
─────────────────────────────────── 
      _progressController.add(DeletionStep.clearingPreferences); 
      await _clearPreferences(); 
 
      _progressController.add(DeletionStep.complete); 
 
    } catch (e, stack) { 
      _progressController.addError(e, stack); 
      debugPrint('[AccountDeletion] FAILED: $e\n$stack'); 
      rethrow; 
    } finally { 
      _progressController.close(); 
    } 
  } 
 
  /// Delete all documents in a Firestore subcollection in batches of 500 
  Future<void> _deleteFirestoreCollection(String uid, String collection) 
async { 
    const batchSize = 500; 
    final ref = 
_firestore.collection('users').doc(uid).collection(collection); 
 
    QuerySnapshot snapshot; 
    do { 
      snapshot = await ref.limit(batchSize).get(); 
      if (snapshot.docs.isEmpty) break; 
 
      final batch = _firestore.batch(); 
      for (final doc in snapshot.docs) { 
        batch.delete(doc.reference); 
      } 
      await batch.commit(); 
 
      debugPrint('[AccountDeletion] Deleted ${snapshot.docs.length} docs from 
$collection'); 
    } while (snapshot.docs.length == batchSize); 
  } 
 
  /// Wipe all user data from every SQLite table 
  Future<void> _clearLocalDatabase() async { 
    final db = await _dbHelper.database; 
 
    // Tables to wipe — ordered to respect foreign key constraints 
    const tablesToClear = [ 
      'user_feedback', 
      'unknown_format_logs', 
      'classification_rules', 
      'sms_transactions', 
      'tax_categories', 
      'linked_accounts', 
      'family_members', 
      'alerts', 
      'recurring_payments', 
      'saving_goals', 
      'transactions', 
      'budgets', 
      'categories', 
      'ce',            // event log table 
    ]; 
 
    await db.transaction((txn) async { 
      for (final table in tablesToClear) { 
        try { 
          await txn.delete(table); 
          debugPrint('[AccountDeletion] Cleared table: $table'); 
        } catch (e) { 
          // Table may not exist in older schema versions — continue 
          debugPrint('[AccountDeletion] Could not clear $table: $e'); 
        } 
      } 
    }); 
  } 
 
  /// Clear all SharedPreferences 
  Future<void> _clearPreferences() async { 
    final prefs = await SharedPreferences.getInstance(); 
    await prefs.clear(); 
  } 
 
  void dispose() { 
    if (!_progressController.isClosed) { 
      _progressController.close(); 
    } 
  } 
}
